# MVP 制作过程问题报告

> **项目**：科研助手 MVP（Python + Colab notebook）
> **日期**：2026-04-04
> **状态**：已完成，但与原始设计存在差距

---

## 一、项目概述

### 1.1 原始目标

基于设计文档，构建一个面向科研工作流的 AI 辅助工具，支持：
- 选题查重与创新判断
- 文献检索与证据聚合
- 数据分析与模型推荐
- 论文提纲与草稿生成
- 结论附证据来源

### 1.2 MVP 实际完成

一个本地运行的 Python 脚本：
```
输入研究问题 → LLM/规则解析 → 模型推荐 → 文献检索 → 生成 Colab Notebook
```

## 二、技术问题清单

### 2.1 环境与依赖问题

#### 问题 1：Python 版本兼容性

| 项目 | 详情 |
|------|------|
| **现象** | `paper-qa>=5.0.0` 要求 Python >= 3.11 |
| **影响** | 用户本地 Python 3.9 不兼容 |
| **解决** | 创建 Python 3.11 虚拟环境 |
| **教训** | 依赖包版本约束应在设计阶段确认 |

```bash
# 解决命令
brew install python@3.11
python3.11 -m venv venv
./venv/bin/pip install -r requirements.txt
```

#### 问题 2：macOS SSL 证书错误

| 项目 | 详情 |
|------|------|
| **现象** | `pip install` 报错 `SSL: CERTIFICATE_VERIFY_FAILED` |
| **原因** | macOS 独立安装的 Python 没有 SSL 证书 |
| **解决** | 添加 `--trusted-host` 参数 |
| **教训** | macOS 特殊环境需要额外处理 |

```bash
./venv/bin/pip install --trusted-host pypi.org --trusted-host files.pythonhosted.org -r requirements.txt
```

#### 问题 3：openalex 包版本不存在

| 项目 | 详情 |
|------|------|
| **现象** | `openalex>=1.0.0` 找不到可用版本 |
| **原因** | openalex 最新版是 0.1.0，版本号命名规则不同 |
| **解决** | 降级为 `openalex`（无版本约束） |
| **教训** | 发布前应验证所有依赖包的可安装性 |

#### 问题 4：paper-qa API 变化

| 项目 | 详情 |
|------|------|
| **现象** | `docs.query()` 方法不存在，报错 `'Docs' object has no attribute 'query'` |
| **原因** | paper-qa 新版本 API 变更为 `aquery()`（异步） |
| **解决** | 重构为异步调用 `docs.aquery()` |
| **教训** | 外部库 API 不向后兼容，需锁定版本或定期更新 |

```python
# 旧版（不工作）
results = docs.query(query, k=top_k)

# 新版（工作）
result = await docs.aquery(query)
```

### 2.2 数据处理问题

#### 问题 5：列名包含换行符

| 项目 | 详情 |
|------|------|
| **现象** | 数据列名包含 `\n`，如 `农药\n使用量\n(吨)` |
| **影响** | 变量名匹配失败，回归报错 `KeyError` |
| **原因** | Excel 原始数据格式问题 |
| **解决** | 标准化时去除换行符和空格 |
| **教训** | 数据入口应做清洗，列名规范化 |

```python
# 原始列名
'农药\n使用量\n(吨)'

# 标准化后
'农药使用量'  # 能匹配
```

#### 问题 6：面板数据年份类型错误

| 项目 | 详情 |
|------|------|
| **现象** | `linearmodels` 报错 `The index on the time dimension must be either numeric or date-like` |
| **原因** | 年份是整数索引，但 linearmodels 需要数值型 |
| **解决** | 显式转换年份为 numeric 类型 |
| **教训** | 时间索引处理是面板数据常见坑 |

```python
# 错误
df_panel = df.set_index(['地区', '年份'])

# 正确
df_panel = df.set_index(['地区', '年份'])
df_panel = df_panel.reset_index('年份')
df_panel['年份'] = pd.to_numeric(df_panel['年份'], errors='coerce')
df_panel = df_panel.set_index('年份', append=True)
```

## 四、关键教训

### 4.1 技术层面

1. **依赖管理**
   - 所有外部包应有明确的版本约束
   - 目标环境的 Python 版本应在项目初期确认
   - 使用虚拟环境隔离项目依赖

2. **数据质量**
   - 数据入口必须经过清洗
   - 列名规范化是必须的预处理步骤
   - 时间/面板数据处理有特殊坑点

3. **外部 API 稳定性**
   - 第三方库 API 可能不向后兼容
   - 需要锁定版本或建立更新机制
   - 必须有 fallback 方案

### 4.2 产品层面

1. **MVP 与架构的关系**
   - MVP 是架构的简化版，不是另一个产品
   - 核心流程和中断点应保留
   - 快速迭代不能牺牲架构合理性

2. **用户体验**
   - 生成的交付物必须与用户环境匹配
   - 错误信息应指导用户如何解决
   - 避免"半成品"感觉

### 4.3 流程层面

1. **设计文档 vs 实现**
   - 偏离设计时应有明确的决策记录
   - 重大变更需要重新评审

2. **问题追踪**
   - 问题应在发生时被记录
   - 事后报告容易遗漏细节
