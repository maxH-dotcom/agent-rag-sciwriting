# 智能科研工作助手：自定义模型与全链路编辑功能架构评估报告

## 1. 背景与目标
为了将系统从单一的“数据处理流”升级为高自由度的“科研工作台”，本次升级计划引入**纯代码片段（Python/R）定义模型**、**前端动态抓取变量**以及**全链路节点可修改干预**的核心能力。

这一升级旨在满足复杂且高度定制化的实证分析需求。例如，在构建碳排放-粮食安全-经济发展 (C-F-E) 的耦合协调度评价体系时，预设的 OLS 或标准固定效应模型往往无法满足多维度指标权重的动态计算。通过开放代码级定义和干预权限，系统将能无缝适应此类复杂面板数据的定制化测算与模型跨数据集迁移。

---

## 2. 核心功能与交互设计

### 2.1 代码驱动的自定义模型构建
* **代码编辑器集成**：前端引入 Monaco Editor，支持 Python/R 的语法高亮与基础代码补全。
* **动态参数抓取 (AST Parsing)**：用户输入核心算法逻辑后（例如一个定义好的 Python 函数），前端点击“解析模型”，请求后端通过静态语法分析（AST）提取出模型运行所需的输入变量（如 `$X$`, `$Y$`, 控制变量等）及超参数配置。
* **动态表单渲染**：前端接收到解析后的 JSON Schema，自动渲染出配置表单面板，供用户填充具体数据列或设定参数默认值。

### 2.2 全链路的中断与修改干预机制
科研数据分析是一个需要不断调优的过程，必须告别“黑盒式”的单向执行。
* **数据映射层修改**：当将基于特定区域（如浙江省）数据跑通的模型，迁移应用到新的数据集时，由于表头命名差异（如 `carbon_emission` 与 `CO2_Total`），在 `data_mapping` 节点必须提供**可视化映射比对视图**，允许用户手动连线或修改变量映射关系，确认无误后再放行。
* **代码沙箱层修改**：在 `code_generation` 或自定义模型执行前，允许用户对 AI 生成的代码或自身历史代码进行二次编辑。
* **结果参数回溯**：如果模型不显著或报错，用户可以点击“回退至参数配置”，修改超参数（如显著性水平、滞后期数）后仅重试当前及后续节点，而非从零开始。

## 3. 后端架构核心要求

为了支撑上述前端交互，后端架构需要进行深度改造，核心挑战在于**动态解析**、**安全沙箱**与**复杂的上下文状态管理**。

### 3.1 静态代码分析与 Schema 生成引擎
后端需要具备“读懂”用户代码的能力，而不是仅仅执行它。
* **Python/R AST 解析器**：需要利用 Python 的 `ast` 模块（或 R 的对应解析包），在不运行代码的前提下，分析出代码片段中的输入参数、预期数据类型以及依赖的第三方包。
* **元数据标准化**：将解析结果转化为标准化的 JSON Schema 并返回给前端，用以动态生成 UI 表单。
* **安全性校验 (Linter)**：在解析阶段即拦截恶意代码（如 `os.system`、文件系统读写等危险操作），而不能等到沙箱执行时才报错。

### 3.2 强隔离的动态执行沙箱 (Secure Sandbox)
这是保障系统稳定性的生命线。
* **环境隔离**：不允许在主宿主机上直接执行用户提交的 Python/R 代码。后端必须引入 Docker 容器化技术或更轻量级的沙箱（如 gVisor、Firecracker）。
* **资源配额管理 (Cgroups)**：限制单次自定义代码执行的 CPU、内存占用量与最大运行时长（Timeout），防止死循环或恶意计算导致后端崩溃。
* **依赖包动态管理**：需要预置常见的科研数据分析包（如 `pandas`, `statsmodels`, `scikit-learn`, `plm` 等），同时支持读取用户设置页中配置的“允许的 Python 包列表”进行动态挂载。

### 3.3 有向无环图 (DAG) 与细粒度状态管理
全链路修改功能要求后端能够精确控制任务的流转状态。
* **图执行引擎**：后端的任务流管理需从简单的线性步骤升级为基于 DAG 的引擎（类似轻量级的 Airflow）。
* **节点级状态持久化**：每一个环节（映射、解析、执行）的输入和输出都需要被独立记录。
* **断点续跑支持**：当用户在某一节点（如模型参数映射）发起修改并提交后，后端需具备使得 DAG 从该修改节点“脏数据清理”并重新向下执行的能力，避免重复执行前面耗时的文献检索或数据预处理节点。

### 3.4 扩展 API 接口设计 (草案)

| 接口路径 | 方法 | 功能描述 | 核心负载 |
| :--- | :--- | :--- | :--- |
| `/api/models/parse` | `POST` | 静态解析自定义代码提取变量 | `code_string`, `language` |
| `/api/models/custom` | `POST` | 保存自定义模型资产 | `name`, `code`, `schema` |
| `/api/tasks/{id}/nodes/{node}/update` | `PUT` | 修改指定节点的数据/参数 | `node_id`, `modified_payload` |
| `/api/tasks/{id}/resume` | `POST` | 从中断或修改处继续执行流程 | `start_node_id` |

## 4. 实施风险提示

1.  **沙箱冷启动延迟**：每次执行隔离代码如果冷启动 Docker 容器，会造成前端长时间 Loading，破坏体验。**建议后端维护一个热备用的沙箱池 (Warm Pool)**。
2.  **代码解析的容错率**：用户写的科研代码往往不够规范，AST 解析极易失败。**建议强制用户按照特定模板函数（如 `def custom_model(data, params):`）的格式书写核心逻辑**，降低后端解析难度。
